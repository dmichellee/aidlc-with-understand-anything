# 4. 운영·정책 문서 추가하기

> 예상 시간: 20분

## 학습 목표

- 코드 밖 지식(운영 문서·개발 정책서)을 Kiro가 활용하게 만든다
- 문서를 `file://` 리소스 / `knowledge` 색인 / 문서 그래프 중 어떤 방식으로 넣을지 판단한다
- 코드 그래프와 문서 지식을 함께 쓰는 워크플로우를 이해한다

## 개념 — 그래프만으로는 부족하다

Code Knowledge Graph는 "코드가 **무엇을 하는가**"는 알려주지만, "**왜 그렇게 해야 하는가**"(비즈니스 규칙·정책·운영 절차)는 코드에 없을 때가 많습니다. 레거시일수록 이 지식이 별도 문서(정책서, 운영 가이드, ERD 설명, 인수인계 문서)에 흩어져 있습니다.

이 문서들을 Kiro의 컨텍스트로 끌어오는 방법은 세 가지입니다.

| 방법 | 언제 쓰나 | 사용 방식 | 특징 |
| --- | --- | --- | --- |
| **`file://` 리소스** | 문서 수가 적고, 항상 참조해야 할 핵심 규칙 | 에이전트 `resources`에 `file://` 경로 추가 | 파일 전체가 매 턴 컨텍스트에 상주. 참조 100% 보장되지만 토큰 비용 큼 |
| **`knowledge` 도구 색인** | 문서가 많고, 필요할 때만 의미 검색 | `/knowledge add ./docs/` 로 색인 | 임베딩 기반 semantic 검색으로 관련 청크만 온디맨드 로드. 토큰 효율적 |
| **문서 그래프** (`understand-knowledge`) | 문서 간 관계·주제 클러스터가 중요 | `/understand-knowledge ./docs/` 로 생성 | 문서를 노드·엣지 그래프로 구조화. 연쇄 참조·충돌 탐지 가능 |

## 실습 0 — 문서를 먼저 마크다운으로 변환하기

이 장의 나머지 실습은 문서가 이미 `.md`라고 가정합니다. 하지만 현실의 정책서·운영 문서는 대부분 **PDF·Word·Excel·한글(HWP)** 형식입니다. Kiro가 잘 활용하려면 먼저 텍스트 기반 마크다운으로 변환하는 것이 좋습니다.

> 왜 변환하나? `file://`·`knowledge` 모두 **텍스트**를 잘 다룹니다. 바이너리 문서(xlsx/docx/pdf)는 표·서식이 깨지거나 검색 품질이 떨어집니다. 마크다운으로 바꾸면 표는 마크다운 표로, 문단은 깔끔한 텍스트로 남습니다.

### 형식별 변환 프롬프트 예시

> **Excel(xlsx) → 마크다운**
> ```
> docs/policies/ 아래의 .xlsx 파일들을 마크다운으로 변환해줘.
> - 시트마다 별도 .md 파일로 나누고
> - 표는 마크다운 표로 변환
> - 원본 파일명 기준으로 docs/policies/md/ 아래에 저장
> ```

> **Word(docx) → 마크다운**
> ```
> docs/handover/*.docx 를 마크다운으로 변환해줘.
> 제목 계층(H1/H2), 목록, 표를 유지하고, 이미지가 있으면 assets/로 빼서 링크로 걸어줘.
> ```

> **PDF → 마크다운**
> ```
> docs/spec.pdf 를 마크다운으로 변환해줘.
> 텍스트 위주로 추출하고, 표는 최대한 마크다운 표로 복원해줘.
> (스캔본이라 텍스트가 안 잡히면 알려줘 — OCR이 필요할 수 있어)
> ```

> **한글(HWP) → 마크다운**
> ```
> docs/policy.hwp 를 마크다운으로 변환해줘.
> hwp5txt 나 hwp 변환 도구를 써서 텍스트를 뽑고 마크다운으로 정리해줘.
> ```

### 변환 도구 힌트

Kiro는 상황에 맞는 CLI 도구를 골라 씁니다. 미리 설치돼 있으면 변환 품질이 좋아집니다.

| 원본 | 자주 쓰는 변환 도구 |
| --- | --- |
| xlsx/docx/pptx/pdf | `markitdown`, `pandoc` |
| docx ↔ 여러 포맷 | `pandoc` |
| xlsx (표 정밀 제어) | Python `openpyxl`/`pandas` |
| pdf (텍스트) | `pdftotext`, `pymupdf` / 스캔본은 OCR(`tesseract`) |
| hwp | `hwp5txt` (pyhwp) |

> 자주 변환한다면 **변환 스킬**(예: `doc2md`)을 하나 만들어 두면 편합니다. "xlsx/docx/pdf가 들어오면 마크다운으로 변환"하는 절차를 스킬로 정의하면, 매번 프롬프트를 다시 쓰지 않아도 됩니다. (스킬 만드는 법은 3장 참고)

> 변환 후에는 **원본과 대조**해 표·수치가 깨지지 않았는지 한 번 확인하세요. 정책서의 숫자(요율·기간 등)는 특히 중요합니다.

## 실습 1 — 핵심 정책을 `file://` 리소스로 상주시키기

항상 지켜야 하는 소수의 규칙(예: 과금 정책, 보안 표준)은 에이전트 `resources`에 직접 넣어 상시 참조하게 합니다.

문서를 정리해 둡니다(예: `docs/policies/` 아래).

```
docs/policies/
├── billing-policy.md # 과금 정책
├── security-baseline.md # 보안 표준
└── data-retention.md # 데이터 보존 규칙
```

에이전트에 연결:

```json
{
 "resources": [
 "file://.kiro/steering/kg-search-rules.md",
 "skill://.kiro/skills/*/SKILL.md",
 "file://docs/policies/**/*.md"
 ]
}
```

> `file://`는 매 턴 컨텍스트에 올라갑니다. 문서가 많으면 토큰이 폭증하므로 **정말 핵심적인 것만** 넣으세요. 나머지는 아래 `knowledge` 방식이 낫습니다.

## 실습 2 — 대량 문서를 `knowledge`로 의미 검색하기

정책서가 수십~수백 개라면 전부 상주시킬 수 없습니다. 이럴 때 `knowledge` 도구로 색인해 두고 **필요할 때만** 의미 기반으로 검색합니다.

> **Kiro에게 시킬 프롬프트**
> ```
> docs/policies/ 폴더의 문서들을 knowledge 지식 베이스로 색인해줘.
> 이름은 "lep-policies" 로 해줘.
> ```

이후 자연어로 검색:

> **Kiro에게 시킬 프롬프트**
> ```
> lep-policies 에서 해지 위약금 계산 규칙을 찾아줘.
> ```

`knowledge`는 임베딩 기반이라 키워드가 정확히 일치하지 않아도 의미가 가까운 문단을 찾아줍니다. 코드 그래프(grep 기반)와 상호보완됩니다.

> **스크린샷 자리** — `knowledge` 도구로 정책 문서를 의미 검색한 결과 (선택)
> <!-- ![knowledge 검색 결과](../assets/04-knowledge-search.png) -->

| | Code Knowledge Graph (grep) | knowledge (임베딩) |
| --- | --- | --- |
| 대상 | 코드 구조·관계 | 문서 내용 |
| 검색 | 키워드/정규식 (+동의어 확장) | 의미 유사도 |
| 강점 | 호출 관계·정확한 위치 | 자연어 질의·개념 매칭 |

## 실습 3 — 문서를 그래프로 만들기 (선택)

문서 간 관계(어느 정책이 어느 도메인에 걸리는지)가 중요하면 `understand-knowledge`로 문서 자체를 그래프화할 수 있습니다.

> **Kiro에게 시킬 프롬프트**
> ```
> /understand-knowledge docs/policies
> ```

## 함께 쓰기 — 코드 + 문서

이상적인 흐름은 다음과 같습니다.

1. **무엇을 바꿔야 하나?** → `knowledge`로 정책·요구 문서에서 규칙 확인
2. **어디를 바꿔야 하나?** → `kg-search`로 관련 코드 위치 특정
3. **어떻게 동작하나?** → `kg-explain`으로 대상 컴포넌트 심층 파악
4. **바꾼다** → 정책(문서)을 근거로, 코드(그래프)를 짚어 변경

> 이 조합이 5장 AI-DLC 워크플로우의 정보 기반이 됩니다.

## 체크포인트

- [ ] 비-마크다운 문서(pdf/word/xlsx/hwp)를 마크다운으로 변환하는 법을 안다
- [ ] 핵심 정책 문서를 `file://` 리소스로 연결했다 (또는 판단해서 제외했다)
- [ ] 대량 문서를 `knowledge`로 색인했다
- [ ] 자연어로 정책 내용을 검색해봤다
- [ ] 코드 그래프와 문서 지식을 언제 각각 쓰는지 구분할 수 있다

## 트러블슈팅

| 증상 | 해결 |
| --- | --- |
| 컨텍스트가 너무 커짐 | `file://`로 넣은 문서가 많은 것. 핵심만 남기고 나머지는 `knowledge`로 이전 |
| xlsx/docx/pdf 정책서 | 먼저 마크다운으로 변환 후 색인 (변환 스킬 활용) |
| `knowledge` 검색 품질이 낮음 | 문서를 주제별로 쪼개 색인, 또는 키워드(BM25) 검색 병행 |
| 문서와 코드가 상충 | 코드가 현행, 문서가 의도. 차이 자체를 리버스 엔지니어링 산출물로 기록 |
